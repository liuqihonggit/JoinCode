## P0: 必须立即统一（架构级不一致）

### P0-1: Namespace 命名统一

**现状**: 5+ 种 namespace 风格并存

| 风格 | 使用项目 | 文件数 |
|------|----------|--------|
| `JoinCode.Xxx.*` | Brain, Hands, Guard, Vault, Eyes, Agents, Llm, Dream, Scheduling, Clock, Composition, Abstractions, Sdk | ~800+ |
| `Core.Bridge.*` | Bridge | ~111 |
| `Core.Xxx.*` | Guard(部分), Clock(部分), Composition(部分), Infrastructure(部分) | ~50+ |
| `IO.Xxx.*` | Infrastructure/IO | ~30+ |
| `McpClient/McpToolHandlers/McpToolRegistry/McpProtocol/Api.Mcp` | Mcp | ~80+ |
| `Plugins.Browser` | Browser | 2 |
| `CodeIndex` | CodeIndex | ~20+ |

**最严重案例**:
- `Mcp` 单项目内部 5 种 namespace: `McpClient`, `McpToolHandlers`, `McpToolRegistry`, `McpProtocol`, `Api.Mcp`
- `Infrastructure` 项目混用 `IO.*`, `Infrastructure.*`, `Core.*`, `JoinCode.Abstractions.*`
- `Guard/Clock/Composition` 项目 csproj RootNamespace 是 `JoinCode.Xxx`，但代码中用 `Core.Xxx`

**统一方案**:
```
规范: JoinCode.{ComponentName}.{SubDomain}
示例:
  Bridge   → JoinCode.Bridge.Client / JoinCode.Bridge.Server / JoinCode.Bridge.Session
  Mcp      → JoinCode.Mcp.Client / JoinCode.Mcp.Protocol / JoinCode.Mcp.ToolHandlers
  Browser  → JoinCode.Browser
  CodeIndex → JoinCode.CodeIndex.Indexing / JoinCode.CodeIndex.Parsing
  Infrastructure → JoinCode.Infrastructure.IO / JoinCode.Infrastructure.Pipeline / JoinCode.Infrastructure.Telemetry
```

**执行步骤**:
1. 先统一 Bridge 项目（111 文件，`Core.Bridge` → `JoinCode.Bridge`）
2. 再统一 Mcp 项目（80+ 文件，5 种 → `JoinCode.Mcp.*`）
3. 统一 Infrastructure 项目（`IO.*`/`Core.*` → `JoinCode.Infrastructure.*`）
4. 统一 Guard/Clock/Composition 中的 `Core.Xxx` → `JoinCode.Xxx`
5. 统一 Browser（`Plugins.Browser` → `JoinCode.Browser`）
6. 统一 CodeIndex（`CodeIndex` → `JoinCode.CodeIndex`）
7. 修复 Infrastructure 中错误写入 `JoinCode.Abstractions.*` namespace 的文件

### P0-3: 错误处理模式统一

**现状**: 3 种模式混用，5 个异常类脱离继承层次

**异常层次断裂**:

```
WorkflowException (基类)
├── ApiException ✅
│   ├── ValidationException ✅
│   ├── AuthException ✅
│   ├── RateLimitException ✅
│   └── ServerErrorException ✅
├── PermissionDeniedException ✅
├── ConfigurationException ✅
├── CodeExecutionException ✅
├── McpProtocolException ✅
├── InstanceCreationException ✅
├── SecurityBlockException ✅
│
├── OAuthException ❌ (直接继承 Exception)
├── HookExecutionException ❌ (直接继承 Exception)
├── ConditionEvaluationException ❌ (直接继承 Exception)
├── VariableResolutionException ❌ (直接继承 Exception)
└── RetryExhaustedException ❌ (直接继承 Exception)
```

**异常 vs Result 混用**:
- `ApiClient` 用 `throw new ApiException(...)` 表示失败
- 上层 `WebToolHandlers` / `TeamToolHandlers` 用 `if (!result.Success)` 判断
- 同一调用链两种模式共存

**统一方案**:

1. **异常层次修复**: 所有自定义异常统一继承 `WorkflowException`
   - `OAuthException` → 继承 `WorkflowException`
   - `HookExecutionException` → 继承 `WorkflowException`
   - `ConditionEvaluationException` → 继承 `WorkflowException`
   - `VariableResolutionException` → 继承 `WorkflowException`
   - `RetryExhaustedException` → 继承 `WorkflowException`

2. **异常 vs Result 边界规则**:
   - **内部层**（组件内部服务间调用）: 使用 `OperationResult<T>` 返回结果
   - **外部边界**（跨组件调用、API 调用）: 使用异常
   - **工具处理器**（ToolHandlers）: 统一使用 `OperationResult<T>`
   - 逐步将 `ApiClient` 的异常模式改为 Result 模式

3. **OperationResult<T> 标准化**:
   - 添加 `#nullable enable`
   - `Data` 和 `ErrorMessage` 属性加 `?` 注解
   - 统一 `Success` + `ErrorMessage` 模式

## P1: 应尽快统一（结构性不一致）

### P1-1: 接口定义位置统一

**现状**: 3 种策略并存

| 策略 | 位置 | 示例 |
|------|------|------|
| Abstractions 项目 | `sdk/Abstractions/XX-domain/` | 90+ 接口 |
| 组件内 `Interfaces/` 目录 | `components/XX/Interfaces/` | Agents |
| 组件内散落实现文件 | 与实现类同目录 | Guard 30+ 接口 |

**问题**:
- Abstractions 内部不一致: `03-hands` 用 `Interfaces/` 子目录，其他领域不用
- `08-transport` 独用 `Contracts` 项目，其他组件用 `Abstractions`
- Guard 30+ 接口内联在实现文件中（如 `IPathValidator` 定义在 `PathValidator.cs` 中）
- 同名接口冲突: `IAgentService` 在 Abstractions 和 Guard 各有一个

**统一方案**:

1. **公开接口** → 统一放 `sdk/Abstractions/` 对应领域目录下，不设 `Interfaces/` 子目录
2. **组件内部接口** → 放在组件内 `Interfaces/` 目录，不散落在实现文件中
3. **消除同名冲突** → Guard 中的 `IAgentService` 重命名为 `IHookAgentService` 或移入 Guard 内部 `Interfaces/`
4. **Transport.Contracts** → 评估是否合并到 `Abstractions/08-transport/`，消除独立的 Contracts 项目
5. **Infrastructure 中的 `Plugins/Interfaces/`** → 移到 `Abstractions/00-core/`

### P1-4: record vs class 统一

**现状**: Result 类型 99 class vs 94 record；Options 类型 37 class vs 30 record

**统一方案**:

| 类型 | 规范 | 理由 |
|------|------|------|
| **Result 类型** | `sealed record` | 不可变值语义，自动获得 Equals/GetHashCode/ToString |
| **Options/Config 类型** | `sealed class` | 需要可变属性（DI 配置绑定），部分需要 `init` 属性 |
| **Request/Response DTO** | `sealed record` | 不可变传输对象 |
| **Entity/State 类型** | `sealed class` | 需要可变状态和封装 |

**需修改的高优先级文件**:
- `Abstractions/.../ToolModels.cs:128` — `ToolResult`: class → record
- `Abstractions/.../OperationResult.cs:3` — `OperationResult<T>`: class → record（需评估兼容性）
- `Abstractions/.../PlanExecutionResult.cs:4` — `PlanExecutionResult`: class → record
- `Abstractions/.../CallToolModels.cs:12` — `CallToolResult`: class → record
- `Abstractions/.../InitializeModels.cs:15` — `InitializeResult`: class → record

### P1-5: 配置注入模式统一

**现状**: 3 种模式并存

| 模式 | 使用情况 | 示例 |
|------|----------|------|
| `IOptions<T>` | 37 处 | `PermissionConfig`, `MemdirOptions`, `QueryEngineConfig` |
| 直接注入配置类 | 更常见 | `WorkflowConfig`, `FileOperationConfig` |

## P2: 应逐步统一（风格级不一致）

### P2-1: Nullable 使用统一

**现状**: 91 处 `!` 抑制运算符

**高频问题**:
- `= null!` 初始化 `init` 属性 → 应改用 `required` 关键字
- `[Inject] private readonly ILogger<...> _logger = null!;` → 源码生成器模式，可接受
- 中间件管道结果 `context.MessageList!` → 应在管道设计中保证非空

**统一方案**:
1. `init` 属性用 `required` 替代 `= null!`
2. `[Inject]` 字段的 `= null!` 保留（源码生成器模式，编译时保证）
3. 管道上下文结果加 `null` 检查或改用 `TryGet` 模式
4. `OperationResult<T>` 添加 `#nullable enable`

| `IBridgeLogger` | Bridge 组件 | 20+ |
| `Console.Write` | JoinCode/ConsoleOutput, TerminalHelper | 30+ |

### P2-3: 事件/回调模式统一

**现状**: EventHandler / Func / Action 三种并存

**统一方案**:

| 场景 | 规范 | 理由 |
|------|------|------|
| **对外公开事件** | `EventHandler<TEventArgs>` | 标准 .NET 模式，支持多订阅 |
| **内部回调/单订阅** | `Func<T, Task>` / `Action<T>` | 简洁，单订阅场景更清晰 |
| **状态订阅** | `IObservable<T>` | 当前未使用，暂不引入 |

**需统一的具体案例**:
- `TransportBase.PayloadReceived` (EventHandler) vs `ConnectionManager.OnMessageReceived` (Func) → 统一为 EventHandler
- `IStore.Subscribe` (Action) vs `IActiveStateService.StateChanged` (EventHandler) → 统一为 EventHandler

### P2-4: 注释风格统一

**现状**: 中英混用 + 部分公开类无文档注释

**统一方案**:

1. **XML 文档注释语言**: 统一使用中文（与 AGENTS.md 规定一致）
2. **公开类/接口必须有 XML 文档注释** — 逐步补全
3. **代码内注释**: 中文为主，技术术语可用英文
4. **禁止删除现有 XML 文档注释** — AGENTS.md 已有规定

---

### P2-5: partial class 使用统一

**现状**: Bridge 全用 vs CodeIndex 全不用

**统一方案**:

| 场景 | 规范 |
|------|------|
| **DI 注册** | 必须用 `partial class`，按注册域分文件 |
| **大文件拆分** | 允许用 `partial class` 按功能域分文件（如 `BridgeMain.Helpers.cs`） |
| **普通业务类** | 不强制使用 `partial class`，单文件即可 |
| **无分文件需求的类** | 不加 `partial`，保持简洁 |

### P3-3: Service 后缀统一

**现状**: 同一职责 4 种后缀混用

| 后缀 | 语义 | 示例 |
|------|------|------|
| `Service` | 无状态业务逻辑 | `WebService`, `NotificationService` |
| `Manager` | 有状态/生命周期管理 | `TokenBudgetManager`, `RemoteClientManager` |
| `Provider` | 数据/实例提供 | `IHttpClientProvider`, `IMcpAuthConfigProvider` |
| `Engine` | 核心算法/执行引擎 | `GoalEngine`, `QueryEngine` |

**统一方案**: 保留当前语义区分，但需明确使用规则:
- **Service**: 无状态、可替换的业务逻辑
- **Manager**: 管理生命周期、持有状态的组件
- **Provider**: 工厂模式，提供实例或数据
- **Engine**: 核心算法、不可替换的执行逻辑

## 风险与注意事项

1. **Namespace 变更是破坏性变更**: 所有 `using` 语句、`extern` 引用、反射调用都需同步修改
2. **每个阶段必须编译验证**: 遵循渐进式开发，每步编译+测试
3. **避免一次性大规模重构**: 按组件逐步推进
4. **partial class 合并需谨慎**: 确保合并后功能等价
5. **record vs class 变更需评估序列化兼容性**: 特别是 `OperationResult<T>` 被广泛使用
6. **异常继承层次变更需同步修改 catch 块**: 确保异常能被正确捕获
